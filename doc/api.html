PrairieLearn API

Overview

There are five main PrairieLearn components:

  1. Database
  2. PrairieLearn API Server
  3. PrairieLearn Web Server
  4. PrairieLearn WebApp
  5. Questions

The PrairieLearn Web Server sends the PrairieLearn WebApp code and resources to the browser. The PrairieLearn Web Server only serves static resources and it does not computation itself.

The PrairieLearn API Server and PrairieLearn WebApp communicate over the Server API (see below).

The PrairieLearn API Server is the only component that communicates with the Database.

Each Question includes both a server and client JavaScript component. The PrairieLearn API Server executes communicates with the server.js Question component. The PrairieLearn API Server also transmits the client.js Question component to the PrairieLearn WebApp as JavaScript source-code. The WebApp then executes and communicates with the client.js component.

The communication between both the Server and the WebApp with the Question occurs over the Question API (see below).

Server API

The PrairieLearn server presents a partially-RESTful HTTP API.

Server API: Identifiers

All ID objects are strings. They should be treated as opaque identifiers and should not be interpreted for any information.

Server API: Method calls

The general pattern is that a collection /collect can generally be accessed in three ways:

  1. A GET to /collect returns all the objects in the collection, each with a minimal representation containing at least the <id> property. An optional query string to the GET may allow for filtering of the return list by certain object properties.
  2. A POST to /collect creates a new object in the collection and returns its <id>. The POST data may need to contain some of the properties of the new object. The newly created object will typically have some of its properties filled in by the server upon creation.
  3. A GET to /collect/<id> returns the single specified object in complete form.

Question API

The PrairieLearn Questions consist of server.js and client.js components. These present a JavaScript API for communication with the PrairieLearn Server and the PrairieLearn WebApp.

The Question components server.js and client.js should be loaded with RequireJS to obtain the corresponding <server> and <client> objects. For example, within Node.js the server Question component can be used like: 

requirejs = require('requirejs');
var serverFile = 'questions/add/server.js';
requirejs([serverFile], function(server) {
  var questionData = server.questionData(variant);
});

Similarly, within client-side JavaScript (with RequireJS loaded), the client Question component can be used like: 

require(['http://<webserverIP>/questions/<qid>/<vid>/client.js'], function(client) {
  client.render(divID, questionData, questionHTML, callbackFcn);
});

Question API: JavaScript code objects

ObjectSpecification
<server> 
{
  "getParams": function(<vid>) {return <params>},
  "gradeAnswer": function(<submittedAnswer>, <params>) {return <grading>}
}
<client> 
{
  "render": function(<divID>, <params>, <questionHTML>, <callback>) {},
  "getSubmittedAnswer": function() {return <submittedAnswer>},
  "showSolution": function(<trueAnswer>) {}
}

Question API: Function specifications

ComponentFunctionArgumentsReturnDescription
<server> getParams <vid> <params> Generate question-specific <params> describing the particular instance of the question using the random <vid>. This function must be deterministic, so that a particular <vid> always generates the same <params>.
<server> gradeAnswer <submittedAnswer> <params> <grading> Determine whether the given <submittedAnswer> object is correct for the question instance corresponding to the given <params>, and return the correctness information and solution in the <grading> object.
<client> render <divID> <params> <questionHTML> <callback> — Render the question into the DOM div element with ID <divID>. The <params> is the JSON object generated by <server>.params() and transmitted via /questions/<qid>/<vid>/params. The <questionHTML> is the String obtained from /questions/<qid>/<vid>/question.html. The final argument is a function with signature callback(<completed>), where <completed> is a Boolean that is true when the user has completed the question, so the question is available for grading.
<client> getAnswerData — <answerData> Construct the <answerData> object from the webpage and return it.
<client> showSolution <trueAnswer> — Display the answer data in the <trueAnswer> object on the webpage.

Sample execution flows

Solving a question with a random variant

The steps to ask and answer a question with a random variant are:

  1. Decide which <qid> we want to attempt.
  2. POST to /qInstances with a partial qInstance of the form: 
    {
  uid: <uid>,
  qid: <qid>
}
  3. The return value will be a completed qInstance object of the form: 
    {
  qiid: <qiid>,
  uid: <uid>,
  qid: <qid>,
  vid: <vid>
}
    Take the randomly generated variant ID <vid> from here.
  4. GET the question HTML and code from /questions/<qid>/<vid>/*
  5. Insert the HTML into the page, then call client.render(), passing a callback to indicate completion.
  6. Wait until the completion callback is called with a true argument and the user indicates the desire to submit the answer for grading.
  7. Call client.getSubmittedAnswer() to obtain a <submittedAnswer> object.
  8. POST to /submissions with a partial submission object of the form: 
    {
  uid: <uid>,
  qid: <qid>,
  vid: <vid>,
  qiid: <qiid>,
  submittedAnswer: <submittedAnswer>
}
  9. The return value will be a completed submission object of the form: 
    {
  uid: <uid>,
  qid: <qid>,
  vid: <vid>,
  qiid: <qiid>,
  submittedAnswer: <submittedAnswer>,
  trueAnswer: <trueAnswer>,
  score: <number>,
}
  10. Show the score to the user and call client.setTrueAnswer(<trueAnswer>) to display the true answer.
